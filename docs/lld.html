4. System Overview

Purpose: Kundli AI is an advanced astrological mentoring platform that blends the mathematical precision of Vedic astrology (computations derived from the Swiss Ephemeris) with spiritual and philosophical counseling based on the Bhagavad Gita (specifically Chapter 16, which covers Divine and Demoniac qualities).

Objectives:

  • Perform high-fidelity sidereal calculations of planetary coordinates, ascendants, houses, yogas, and doshas based on user-supplied birth coordinates.
  • Maintain user context across browser refreshes and server restarts without requiring explicit email/social sign-ups.
  • Implement semantic text retrieval (RAG) using highly efficient, dependency-light local key-term embeddings and ChromaDB.
  • Structure prompt payloads logically in a designated block hierarchy before querying LLMs to guarantee high coherence and alignment with spiritual guidelines.

Scope: Computes birth charts, handles conversational Q&A under a warm, empathetic mentoring persona, runs local text searches over selected scriptures, and caches user profiles. Future expansions like complete multi-lingual support, voice integration, and full database persistence are scoped separately.

Technology Stack & Core Dependencies:

  • Frontend: React 19 (SPA), TypeScript, Tailwind CSS, Vite.
  • Backend: FastAPI (Python 3.11), Uvicorn.
  • Astrology Engines: Swiss Ephemeris (`pyswisseph` binding).
  • Vector Databases: ChromaDB (`chromadb` client).
  • LLM Connectors: Direct HTTP REST calls to Groq API (`llama-3.3-70b-versatile` model) with offline Claude prompts for safety fallback.

6. Module Breakdown

Module: Services / Astrology

Purpose: Computes astronomical placements and translates them into Vedic interpretations (zodiac signs, nakshatras, yogas, doshas).

Responsibilities: Bind with Swiss Ephemeris files, compute planetary angles, map houses, evaluate combinations, and identify afflictions.

  • Input: Year, month, day, hour, minute, second (UTC), latitude, longitude, and timezone offset.
  • Output: Computed horoscope dictionary containing metadata, planets, houses, yogas, and doshas.
  • Dependencies: `pyswisseph` (low-level binding).
  • Core Functions: `calculate_horoscope_data(...)`
Module: Services / RAG

Purpose: Embeds, indexes, and queries religious scriptures (specifically Bhagavad Gita) for contextual text injections.

Responsibilities: Page extraction, sliding window chunking, computing key-term vector matrices, and executing database queries.

  • Input: Raw text queries, raw PDF/JSON document files.
  • Output: List of top match text strings.
  • Dependencies: `chromadb`, `pypdf`.
  • Core Classes: `RAGPipeline`, `ChromaStore`, `Embedder`, `CharacterChunker`, `PDFDocumentLoader`, `Retriever`.
Module: Services / Prompts

Purpose: Formulates complete prompts based on structured contexts for ingestion by the active LLM client.

Responsibilities: Synthesize system rules, format history logs, detect query intent, inject relevant scriptural verses, and compile the final block sequence.

  • Input: Raw query string, chart coordinates, user profile, chat history list, retrieved RAG passages.
  • Output: Consolidated prompt string.
  • Dependencies: `services/astrology`, `services/memory`.
  • Core Functions: `assemble_unified_prompt(...)`, `is_financial_query(...)`.
Module: Services / Memory

Purpose: Manages in-memory chat sessions and coordinates filesystem storage for persistent users.

Responsibilities: Thread-safe profile reading/writing (JSON), session creation, chat history formatting, and caching computed natal data.

  • Input: UUID strings, message payloads, computed chart dictionaries.
  • Output: Loaded profile payload, formatted history logs, success status booleans.
  • Dependencies: standard file I/O, `threading.Lock`.
  • Core Classes: `ProfileStore`, `SessionStore`, `ChatHistory`.

7. Backend Architecture

The backend of Kundli AI is structured around a decoupled architecture designed to handle fast calculations, vector store lookups, and session management in a stateless hosting model:

1. API Layer (Routers)

Managed by FastAPI. It exposes specific JSON endpoints and maps parameters to typed Pydantic models. Routers are split by domains (astrology computation, conversational chat, and profile management).

2. Business Logic & Astrology Service

Calculates planetary positions on the sidereal zodiac (using the Lahiri Ayanamsa). It evaluates house boundaries (using the Whole-Sign house system), nakshatra bounds, yoga alignments, and active doshas (like Sade Sati, Manglik, and Kaal Sarp).

3. Retrieval-Augmented Generation (RAG) Service

Runs a custom vector-space search. It uses an occurrence frequency matrix over a predefined vocabulary of spiritual keywords, bypassing heavy deep-learning dependencies to maintain a fast, lightweight deployment runtime. Results are sorted based on cosine similarity scores.

4. Session & Memory Manager

Uses a dual storage layout:

  • Transient Session Cache: RAM-based dictionary tracking active chat logs per tab session.
  • Persistent Profile Store: Disk-based JSON files linked to an anonymous client-side UUID, ensuring natal charts are persisted across server lifecycles.

  • Geocoding Resolution: Uses OpenStreetMap Nominatim APIs to convert plain text city inputs into geographic coordinates (latitude and longitude) for precise horoscope calculations.

10. Database / Storage Design

Kundli AI uses a hybrid storage model designed to balance retrieval performance, filesystem caching, and session lifecycle requirements:

Vector Database (ChromaDB) Collection Schema

Scriptures are indexed in a local ChromaDB collection named `gita_ch16`. Each document chunk is stored with the following metadata fields to support precise matching:

  • document_id: Unique identifier (`gita_ch16_chunk_{idx}`).
  • document_content: Raw Sanskrit text, translation, and significance comments.
  • metadata: Includes `source` (filename), `page_number`, and `chunk_index`.
  • vector: Generated embedding array of length 39 (based on key-term vocabulary frequencies).

JSON File Storage Schema

Persistent user profiles are stored as JSON files on the server's filesystem under `backend/data/profiles/{uuid}.json`.

11. RAG Pipeline

The scripture context retrieval workflow follows a structured pipelined approach to process queries and inject matching references:

flowchart TD A[Gita Chapter 16 Source File] --> B[PDF/JSON Document Loader] B --> C[Character Sliding Chunker] C --> D[Keyword Embeddings Generator] D --> E[Local ChromaDB Store] F[User Chat Message] --> G[Query Engine Preprocess] G --> H[Keyword Embeddings Generator] H --> I[Retriever Query search] E -.-> I I --> J[Prompt Builder Context Injection] J --> K[Groq LLM API Request] K --> L[Empathetic Answer Response]

Seeding: On application startup, the `RAGPipeline` checks the vector store count. If empty, the document loader parses Chapter 16 of the Bhagavad Gita, splits it into chunks of 500 characters (with a 50-character sliding overlap), generates embeddings, and saves the vectors to disk.

Querying: When the user submits a message, the system normalizes the query text, generates its embedding vector, performs a cosine-similarity search, and extracts the top 2 matching verses to inject into the LLM prompt context.

12. Horoscope Pipeline

Horoscope data flow is divided into a static calculation phase and a dynamic calculations phase to balance calculation accuracy with performance:

flowchart LR A[Birth Details Form] --> B[Swiss Ephemeris Calculator] B --> C[Planetary Longitudes D1] C --> D[Zodiac Houses Map] C --> E[Nakshatra / Pada Math] D & E --> F[Yoga Combinations Evaluator] D & E --> G[Natal Doshas Evaluator] F & G --> H[Natal Chart Cache saved] I[Current Transit Saturn] --> J[Sade Sati Transit check] H & J --> K[Compiled Prompt Synthesis]

The static calculations compute the birth positions (Lagna sign, Nakshatras, and natal Yogas/Doshas like Manglik or Kaal Sarp). The dynamic calculations compute transit-dependent aspects on-demand (such as Sade Sati, which is calculated based on current Saturn transits relative to the natal Moon position).

13. Prompt Compilation Pipeline

Before querying the LLM, the backend compiles system rules, session data, and scripture context into a structured system prompt:

  1. Conversation History: Inserts the last 5 chat turns to maintain conversational continuity.
  2. Conversation Summary: Adds a short summary of the previous assistant turn to keep the context size efficient.
  3. Intent Detection: Identifies query topics (e.g. Wealth, Career, Spirituality) to route queries (e.g., swapping to the specialized 18-section Financial System Prompt if wealth-related).
  4. User Profile: Injects birth coordinates and metadata.
  5. Horoscope Context: Appends the ascendant, moon sign, nakshatras, house lordships, and planetary degrees.
  6. Yogas & Doshas: Injects identified astrological combinations.
  7. Retrieved Gita context: Appends matching passages from the vector store search.
  8. User Question: Appends the active query.

17. Request Lifecycle

Every chat request from the browser to the backend and back follows a structured step-by-step lifecycle:

  1. Dispatch: The client triggers an asynchronous fetch request (`POST /api/chat`) containing the message payload and session tokens.
  2. Session Verification: The FastAPI backend intercepts the request and verifies the session token. If the session has expired, it attempts to restore the profile using the persistent user UUID.
  3. Intent Analysis: The system pre-processes the query to identify search intents (e.g. classifying "wealth" keywords to select the specialized financial prompt).
  4. Vector Store Query: The RAG engine calculates text query vectors, queries ChromaDB, and returns the top 2 matching verses from the Bhagavad Gita database.
  5. Prompt Compilation: The prompt builder synthesizes system personas, conversation history, user profiles, horoscope data, and retrieved scriptures.
  6. Inference Request: The backend dispatches the compiled prompt payload to the Groq API endpoint.
  7. Response Delivery: The server receives the text response, appends the turn to the session history cache, and returns the markdown response to the client UI.

19. Session Management

Session state is synchronized across frontend and backend environments:

  • Persistent UUID: Generated via `crypto.randomUUID()` on first visit and stored in the browser's `localStorage` as `kundli_user_id`.
  • Transient Session ID: Generated as a random alphanumeric string on page load to track separate conversations in the in-memory backend cache.
  • Restore Flow: On page refresh, the client queries `/api/profile/{uuid}`. If found, the backend restores session state and the UI transitions to the ready state, skipping form wizard steps.
  • Manual Resets: Clicking "Update Birth Details" sends a `DELETE` request, removes the persistent profile, clears session variables, and resets the UI state.

20. Security Specs

Security protections are implemented across the stack:

  • Input Validation: FastAPI uses Pydantic schemas to validate and sanitize incoming request parameters.
  • Zodiac Calculation Sanity Checks: Coordinates are validated to ensure latitudes fall within `[-90, 90]` and longitudes fall within `[-180, 180]`.
  • API Key Isolation: LLM keys are restricted to backend environment variables, shielding keys from client exposure.
  • CORS Policy: FastAPI CORS middleware restricts API access to authorized domains (e.g. localhost during development, production host on Render).
  • Prompt Formatting: Prompts are compiled inside structured block templates to prevent prompt injection attempts.

21. Deployment Architecture

Kundli AI is deployed in a unified production configuration on Render:

  • React Client Page: Hosted as a Render Static Site, building from the React frontend folder.
  • FastAPI Web Application: Deployed as a Render Web Service running Uvicorn.
  • Database Persistence: ChromaDB vector stores and persistent profile JSON files are written directly to local folder paths (`backend/chroma_db/`, `backend/data/profiles/`).
  • Storage Limitations: Since Render containers use ephemeral filesystems, files will reset on container restarts. This is noted in the roadmap to be migrated to cloud databases during the scaling phase.

23. Design Decisions

Design Decision: Why FastAPI instead of Django/Node?

FastAPI provides high-performance asynchronous request handling and automatic OpenAPI documentation generation. It features native integration with Python's scientific ecosystem (e.g. `pyswisseph` bindings), making it a natural choice over Node or Django.

Design Decision: Why ChromaDB local persistent files?

ChromaDB offers a lightweight vector database that runs embedded in python processes. By storing indexes locally, we avoid the overhead and costs of managing external cloud vector services for single-chapter scripture searches.

Design Decision: Why direct REST calls for LLM instead of LangChain?

Direct HTTP requests reduce library dependencies, speed up application startup times, and provide complete control over prompt payloads without the complexity of heavy frameworks.

24. Known Limitations

  • Scripture Coverage: The local vector store search is currently limited to Chapter 16 of the Bhagavad Gita. General queries fallback to default Vedic guidance.
  • Disk Reset: Stored profile files are local. If the Render web service container restarts, local user profiles will clear unless persistent volume mounts are configured.
  • Local Embeddings: The embedding system uses a custom term-frequency vectorizer. High-level conceptual matching (semantic synonyms) is limited compared to dense Transformers, but operates at virtually 0ms local compute overhead.

25. Conclusion

The Kundli AI platform features a robust, decoupled architecture that pairs sidereal Vedic computations with local semantic retrieval. By implementing client-side UUID persistence, the application matches modern user experience patterns by eliminating the friction of manual logins. The structured layout establishes a clear roadmap for scaling the database, storage, and agent components in future phases.
